# 微信公众号文章排版 Rubric，基于样本第一版

## 样本来源

参考目录：

```text
docs/reference/wechat-layout-samples/
```

当前样本数：13 篇。

## 核心结论

公众号排版不是把每句话都拆成一段。高赞文章常见节奏是：

```text
短段落开场
-> 2 到 4 句组成一个推进段
-> 图片或截图打断
-> 关键判断独立成段
-> 小标题只在教程/技术拆解/方法论场景使用
-> 每一屏都有视觉锚点
```

## 段落规则

### 开头 300 字

- 允许大量短段。
- 目标是快速建立场景、冲突或好奇心。
- 3 到 6 段内必须让读者知道“这篇到底在聊什么”。

推荐结构：

```text
具体事件
反常点
作者反应
这篇文章要聊的真正问题
```

### 中段推进

不要全篇一行一段。中段应该混合三种段落：

- 短停顿段：1 句，用来制造节奏。
- 推进段：2 到 4 句，用来讲清楚一个动作、判断或案例。
- 信息段：4 到 6 句，只在解释机制、流程、教程步骤时使用。

如果连续 8 段都是 1 句，读者会感觉被“碎片轰炸”，不是留白。

### 关键判断

关键判断可以独立成段，例如：

```text
这不是文风问题。

这是验收标准问题。
```

但独立段必须少而准。每 500 到 800 字出现 1 到 3 个即可。

## 视觉锚点

样本里图片、截图、链接卡片很常见。技术/工具/产品类文章尤其需要视觉锚点。

规则：

- 开头 800 字内最好出现第一个图片或截图位。
- 每解释一个真实操作，配一个截图位。
- 如果没有真实图片，先标注 `![图片]`，后续再补。
- 图片前后要有一句解释，不要让图片孤零零出现。

## 小标题规则

不是所有公众号文章都要小标题。

适合小标题的场景：

- 技术拆解
- 教程步骤
- 方法论分享
- 多个功能逐一展示

不适合小标题的场景：

- 叙事型故事
- 情绪型观点
- 单一主线的复盘

如果使用小标题，建议用自然标题，不要报告式标题：

```text
**一，先给 AI 一张工单**
```

而不是：

```text
## 系统架构设计
```

## 加粗规则

加粗只用于：

- 一句话结论
- 小节标题
- 读者必须记住的判断

不要整段加粗。

## 列表规则

列表可以用，但要少。

适合用列表：

- 命令清单
- 步骤
- 对比项
- 三到五个明确要点

不适合用列表：

- 把正文观点全部 bullet 化
- 连续多个列表堆叠

## 当前 PTCE 文章适用模板

这篇文章是“产品方法论 + 技术实现观察”，适合采用：

```text
开头故事
-> 问题抽象
-> 三个核心动作逐一展示
-> 内容类型和质量标准解释
-> MVP 边界
-> 更大的判断
```

排版要求：

- 开头短。
- 中段使用小标题分块。
- 每个小标题下 3 到 6 个自然段。
- 插入 3 到 5 个图片占位。
- 保留少量短句断裂，不要全篇断裂。

## 接入 PTCE 写作体系

这份 rubric 不是一次性参考文档，而是 `public_article` 的 WeChat 输出检查规则。

接入点：

- `public-article-writing` 在生成或改写 WeChat 长文时必须读取并执行本 rubric。
- `ptce-content-writing` 在保存 `public_article` 的最终 `draft` artifact 前，必须先完成 WeChat layout pass。
- `WorkflowProfile.public_article.default` 包含 `wechat_layout_check` artifact contract，用于在 output package 中暴露排版检查环节。

执行顺序：

```text
appeal_brief
-> reader_resonance_check
-> evidence_bedrock
-> narrative_outline
-> draft
-> voice_pass
-> wechat_layout_check
-> truth_check
-> publication_package
```

最低验收：

- 禁用词和禁用标点扫描无命中。
- HKR 已通过，尤其是 R，共鸣不是最后润色，而是选题和结构的前置约束。
- 语气贴合检查已完成，关键判断有短句抓手，但没有复制某个作者的人设或固定尾部。
- 中段不是全篇一行一段。
- 需要技术/产品解释时，有小标题和视觉锚点。
- `![图片]` 占位数量符合内容密度，不少于 3 个，不滥放。
