# 我是怎么用 AI/agent 把 AI Homework Review 做出来的

> 如果你对 vibecoding 感兴趣，但还没有真的把它用进一个持续演进的项目里，那我建议你少看一点“一句话生成一个应用”的演示，多看一点项目是怎么在真实约束里往前拱的。

我最近一直在用一个项目验证这件事。这个项目叫 `AI Homework Review`，最初只是一个移动端优先的 AI 作业批改 MVP，目标并不夸张：让老师打开页面、填入标准答案、上传答题卡图片，然后拿到一个真实的批改结果。

后来看起来最简单的这条路，反而把真实项目里会遇到的问题都带出来了：上传不是上传按钮能点就算完成，模型不是接口能返回就算接通，批量批改也不是把单次请求放大十倍就自然成立。再往后，项目还一路长到了 `batch review`、离线任务中心、MySQL 状态存储、常驻 worker、移动端任务导航和部署收口。

回头看这个过程，我最大的感受不是“AI 帮我省了多少代码”，而是我终于摸清了一件事：**agent 最适合做的，不是替我负责项目，而是替我推进那些已经被定义清楚的部分。**

## 先说结论

如果你只想先拿走最重要的一句话，那就是：

> vibecoding 真正有用的前提，不是你会不会写 prompt，而是你能不能把项目拆成 AI 真能接得住的粒度。

在 `AI Homework Review` 这个项目里，agent 确实帮我加快了很多工作：

- 把空白项目迅速推到第一版可运行状态
- 沿着既有模式补页面、接口、共享类型和测试
- 把已经明确的重构任务批量搬完
- 在 spec 已经确定的前提下，持续推进一条比较长的实现链

但它也很稳定地暴露了另一面：

- 一旦需求边界没收紧，它会把你的模糊也一起“合理实现”
- 一旦进入部署、权限、状态一致性、异常恢复和排障，人必须重新回到主驾驶位
- 你如果只靠 prompt 推项目，短期会很爽，后期会非常容易返工

这篇文章我不想写成“AI 真厉害”的鸡血稿。我更想讲清楚，**我是怎么一步步把 agent 用顺手的，以及它到底在哪些地方帮上了忙，哪些地方绝对不能交给它。**

## 第一阶段：2026-04-06，先把项目推过“空白期”

这个项目在 2026 年 4 月 6 日最早定下来的目标很克制。

不是完整教学平台，不是老师后台，不是学生体系，而是一个能直接展示产品价值、同时又能让人真的体验一次批改流程的站点。首页既是产品入口，也是 demo 入口。项目范围也限定得很死：先支持固定版式数学答题卡，先支持选择和填空，先把“上传 -> 识别 -> 判分 -> 讲评 -> 返回结果”这条链路跑通。

这时候 agent 最有价值。

因为空白项目最难的，不是某一段代码写不出来，而是你很容易一直停留在“我知道要做什么，但还没有一个能跑的起点”。前端页面骨架、演示区表单、API 骨架、共享类型、基础测试，这些工作都很适合交给 agent 去产出第一版。它的作用不是替我思考，而是把项目快速推过那个最耗人耐心的启动阶段。

但从第一天开始，我就没把它当成“随便写点 prompt 看看会发生什么”。这个项目一开始就有设计文档、范围约束和部署方向。也就是说，我不是先让 agent 写，再看写成什么样；而是先把边界说清楚，再让它在这个边界里推进。

这是我后来越来越确信的一点：**AI 最怕的不是难题，而是没有边界的问题。**

## 第二阶段：2026-04-11 到 2026-04-12，从“会动的 demo”到“真实链路”

真正让我开始修正自己对 vibecoding 理解的，是 4 月 11 日到 4 月 12 日这一段。

一开始，最容易做出来的是“像真的一样”的东西。页面能点，流程会跳，结果页能展示一段看起来像 AI 讲评的文案。这类 demo 在演示时很讨巧，但它其实绕开了最难的部分：图片到底有没有真正上传，对象存储和权限有没有真正配通，OCR 到底有没有真的读到图，文本模型返回的内容是不是来自真实推理链路。

所以我开始把项目往“真链路”上推。

最后落成的结构是：

- 前端放在 `GitHub Pages`
- 后端放在 `阿里云函数计算`
- 图片先走 `OSS`
- 前端优先通过 `STS` 临时凭证直传
- 服务端读图后调 `qwen-vl-ocr-latest`
- 讲评生成走 `gpt-5.4`

这时候，agent 依然有用，但用法已经和第一阶段不同了。

它可以很快帮我把一条我已经想清楚的链路补齐，比如：

- 增加上传策略接口
- 串起前端上传流程和后端读图流程
- 补 OSS 直传失败后的代理上传回退
- 补配置项、测试和部署文档

但真实的困难不在“写几个接口”，而在工程细节。

比如我在这一段里真正花时间的，不是页面怎么布局，而是这些问题：

- `RAM 用户` 和 `RAM 角色` 到底谁负责 `AssumeRole`，谁负责 `PutObject`，谁负责 `GetObject`
- `OSS endpoint`、bucket 域名、签名 URL 在不同阶段能不能混写
- 浏览器直传如果被 CORS 或预检卡住，产品层面要不要整体降级
- OCR 如果返回 `"1"` 而不是 `1`，后端 schema 是该报错还是该兼容

这些问题很能说明 agent 的真实边界。

它能给出方案，能补实现，能加保护层，能很快展开几种选择。但它不能替我决定，**这个项目此刻最重要的是“技术最优”，还是“体验先过验收”。**

最后我做的决定并不浪漫，但非常工程化：浏览器优先走 `STS + OSS` 直传，失败就自动回退到后端代理上传。不是因为这条路最优雅，而是因为真实产品不能让老师卡死在上传这一关。

这类判断，最终还是得人来做。

## 第三阶段：2026-04-13，项目一复杂，纯 vibe 就不够了

如果项目停在单张答题卡批改，其实还不算真正进入复杂区。

复杂度真正开始抬起来，是我在 2026-04-13 把项目往 `batch review` 推的时候。

目标不再是“上传一张图片，返回一个结果”，而是：

- 上传整班同题 PDF
- 再上传评分标准材料
- 对每一页学生答案做逐页批改
- 返回页级评论和班级汇总

一旦需求变成这样，系统结构也跟着变了：

- 输入从单张图片变成了 PDF 和多份材料
- 处理单位从“一次请求”变成了“一个任务中的多页”
- 结果不再只是一个分数，而是任务状态、页级反馈、失败页、重试流转

如果还用最早那种“想到什么就让 AI 写什么”的方式，项目很快就会失控。因为这时候任何一个模糊点，都会被 agent 擅自补全成“某种实现”，而这种实现很可能只是在局部自洽，并不符合整个项目的长期结构。

所以从这一阶段开始，我对 agent 的使用方式变了。

我不再直接让它“做一个批量批改功能”，而是先把工作压成更清楚的层次：

1. 先写 spec，明确这次只扩哪一层，不动哪一层。
2. 再写 plan，把大任务拆成若干个能独立完成的小任务。
3. 再让 agent 沿着 plan 去补共享 contract、provider、页面、轮询状态和结果页。
4. 每走完一段，就用测试和 review 把它收口。

这时候 agent 的效率反而更高了。

因为它最擅长的，其实不是“凭空发明复杂系统”，而是：

- 基于既有约束补一层新接口
- 把一个过载页面拆成几个更明确的 route
- 沿着现有模式补测试和回归用例
- 在清楚的 ownership 下完成一段局部实现

这也是我第一次真切感受到：**agent 不是越自由越强，反而是边界越明确越强。**

## 第四阶段：2026-04-24 到 2026-04-26，真正的难点变成工程收口

到了 4 月 24 日之后，这个项目已经不再是“继续加功能”那么简单了。

这一段我更关心的，是怎么把已经长出来的系统收紧。

### 2026-04-24：统一模型调用，不再到处散着请求逻辑

我开始把单张批改和批量批改背后的模型请求往一个更统一的客户端上收。

原因很简单：如果 OCR、讲评、批量多模态评分都各自维护一套请求执行方式，那后面遇到超时、有限重试、退避、结构化日志时，你会在很多地方重复修同一类问题。表面上只是“代码有点散”，实际上会直接抬高后续排障和替换模型的成本。

这一层很适合 agent 参与，因为重构目标很明确：不改前端 contract，不改大体行为，只把请求执行能力往共享客户端收。

### 2026-04-25：把批量批改从请求内执行，改成真正的离线任务系统

这是一个非常关键的分水岭。

之前批量批改虽然能跑，但本质上还带着很强的“请求内推进”味道。项目一旦更真实，你就很难继续把大任务都塞进一次 API 请求里。于是我开始把这条链路往离线任务系统改：

- API 只负责创建和查询任务
- `MySQL` 保存任务状态
- 常驻 `worker` 在后台逐页执行
- 前端的 `#/batch-review` 不再只是上传向导，而是一个任务中心

这个阶段最能说明一个事实：**当问题变成状态流转、可靠性和运维结构时，人必须明显介入得更多。**

agent 仍然很好用，但它主要负责的是：

- 补 repository 和 contract
- 拆页面
- 接轮询
- 补任务详情和通知页
- 沿着既定结构把实现铺开

至于“到底为什么要从 inline 切到 offline”、“任务和 worker 的边界怎么划”、“哪些信息应该由 API 持有，哪些应该落到持久层”，这些决定都不能外包。

### 2026-04-26：开始处理只有真实项目才会暴露的问题

最近几次提交最说明问题。

它们已经不是“再加一个新功能”，而是这些很现实的收尾工作：

- 函数计算访问 MySQL 需要挂到 VPC
- 离线 worker 相关环境变量要完整暴露到部署配置
- 异步任务路由要和实际执行器重新对齐

这种问题没有任何演示效果，但它们决定了系统到底是“看起来长大了”，还是“真的长成了”。

也是到这个阶段，我对 vibecoding 有了一个更稳的判断：

> 当项目还在长骨架时，agent 是加速器。  
> 当项目进入状态一致性、部署约束和线上排障时，人必须重新成为主驾驶。

## 在这个项目里，agent 真正帮上忙的地方

如果让我只挑最有价值的几个点，我会列这几个：

- 把空白模块快速推到第一版可运行状态
- 沿着既有代码风格补接口、补页面、补共享类型
- 在既有 contract 之上批量补测试
- 按明确边界做重构搬运
- 把 spec、plan、实现、验证串起来，降低上下文切换成本

这些工作看上去不那么“酷”，但它们才是一个项目能持续往前跑的燃料。

## 在这个项目里，我不会交给 agent 的事情

同样明确的是，我现在有几类事情不会交给它：

- 产品边界判断
- 架构边界判断
- 可靠性优先级判断
- 最终的质量收口
- 真正的线上排障

不是因为它完全做不了，而是因为这些判断一旦做错，责任最后还是我来兜。

你可以让 agent 给你三个方案，但你不能把“哪一个方案值得承担后果”这件事也外包出去。

## 如果让我重来一次，我会更早从“prompt 驱动”切到“流程驱动”

这是这次项目给我的最大复盘。

以前我会觉得，项目刚开始的时候没必要太早写 spec、plan、review，先让 AI 帮我冲起来再说。现在我反而觉得，越早引入 agent，越应该越早把流程固定下来。

因为 agent 最大的问题不是能力不够，而是推进速度太快。它一旦开始真的帮你产出，项目就很容易在几天里长出很多东西。没有流程，增长本身就会变成新的混乱源。

所以如果你今天也想开始自己的 vibecoding 项目，我的建议非常具体：

1. 先找一个真实问题，不要找一个只适合录 demo 视频的题材。
2. 第一阶段允许 agent 多写，但前提是范围已经被你定义清楚。
3. 一旦进入真实链路、外部服务和异步状态，立刻切到 `spec + plan + verify` 驱动。
4. 永远把最终的产品判断、架构判断和质量判断抓在自己手里。

我现在已经不太相信“一把梭的 vibe”了，但我比以前更相信另一件事：**只要你能把问题说清楚，把边界立住，把流程搭起来，AI 和 agent 的确能把一个真实项目推得很快。**

`AI Homework Review` 对我来说，最大的价值不是它证明了 agent 会写代码，而是它让我确认了一种更可靠的协作方式。不是我把项目交给 AI，而是我把项目拆到 AI 真正接得住的粒度，然后让它在那个范围里尽可能快地推进。
