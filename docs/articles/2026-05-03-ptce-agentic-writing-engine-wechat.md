# 我给 AI 写作装了一个工单系统

事情是这样的。

这两天我在做一个很小的东西。

小到从外面看，甚至有点无聊。

它不是新的写作模型，也不是新的 Prompt，更不是那种输入一个标题，三秒钟给你吐出一篇爆款文章的玩意。

我做的是一个写作工单系统。

对。

工单。

就是那种听起来很土，很不性感，很像公司内部系统的东西。

但我做着做着，突然有点上头。

因为我发现，AI 写作现在最缺的，可能不是更会写，而是更知道自己到底在干嘛。

![图片]

你应该也遇到过这种情况。

你跟 AI 说，帮我写一篇文章，它写了。你说，帮我写一份 PRD，它也写了。你说，帮我写一份技术文档，它还是写了。

而且每一篇都挺顺。

顺到你一开始甚至会觉得，哎呦，不错哦。

但你多看几次就会发现不对劲。

它写公众号的时候，像在写咨询报告。它写 PRD 的时候，像在写产品宣传稿。它写技术文档的时候，像在写一篇很有条理但不一定能跑的说明文。

它什么都会写。

但它经常不知道这篇东西真正怕什么。

公众号文章最怕没人读下去。PRD 最怕团队看完还是不知道要做什么。技术文档最怕读者照着做，结果掉坑里。普通说明最怕绕来绕去，读者只记得你说了很多。

这四件事都叫写作。

但它们根本不是同一个工种。

**一，先给 AI 一张工单**

所以我这次干的事，就是给 AI 写作分工。

项目里现在有四个 writing skill，`public-article-writing` 负责公开文章，`prd-writing` 负责产品需求文档，`technical-doc-writing` 负责技术文档，`general-writing` 负责通用写作。

每次任务只能选一个主 skill。

这点我现在觉得特别重要。

因为很多 agent 系统特别喜欢把所有能力都塞给模型。你既要它有公众号的读者拉力，又要它有 PRD 的可执行性，还要它有技术文档的准确性，再顺便有一点个人风格。

听起来很全能。

写出来就很怪。

像一个人穿着西装打篮球，手里还端着一碗面。

不是不能动，就是怎么看都别扭。

我现在更相信，一个写作任务应该先确定身份。你现在是写公众号，那你就先考虑读者愿不愿意往下读。你现在是写 PRD，那你就先考虑团队能不能拿它开工。你现在是写技术文档，那你就先考虑它跟代码、接口、配置是不是对得上。

这不是文风问题。

这是验收标准问题。

于是项目里加了一个东西，叫 `ContentTask`。

这个名字听起来很工程，但你可以把它理解成，给每一次写作开一张工单。

这张工单里会记录几件事，写什么类型，写给谁，为什么写，用哪套 skill，按哪套标准检查，最终要交付哪些东西。

我第一次把它跑起来的时候，终端里出现了一大段 JSON。

说实话，一点都不美。

但我看得还挺爽。

![图片]

因为那里面不只是一个任务 id。

它还把 `workflowProfile`、`qualityRubric`、`skillBinding`、`outputPackage` 一起返回了。

这玩意很像系统递给 agent 的一张任务卡。你不是来随便写一段文字的，你是来完成这个任务的。

任务类型是什么，可以做哪些动作，要产出哪些 artifact，完成之前要看哪些 checklist，全部写在卡上。

这种感觉，跟普通聊天窗口完全不一样。

普通聊天窗口里，AI 写完一段话，就结束了。你觉得好就复制走，你觉得不好就继续改。但系统其实不知道发生了什么。

它不知道这篇东西属于哪类内容，不知道它现在是草稿，还是待 review，还是已经完成。也不知道哪一版是最终稿，更不知道这篇东西到底有没有被保存成一个可追踪的产物。

说难听点，很多 AI 写作产品现在就是这样。

它像一个特别会说话的临时工。

来得快。

走得也快。

写完之后桌上留下一张纸。

至于这张纸是谁要的，按什么标准验收，后面谁来改，改完算不算完成。

不知道。

我想做的，是给这个临时工配一个工位。

甚至不是工位。

是工牌。

**二，三个命令，把聊天变成交付**

你来干活可以。

但你得进系统，得领任务，得交付产物，得留下记录。

所以现在 CLI 里有三个命令，我越看越喜欢。

`content create`，开工单。

`content artifact add`，交付产物。

`content complete`，标记完成。

就这三个动作。

特别朴素。

但它把 AI 写作从一场聊天，变成了一次可追踪的工作流。

![图片]

比如你让 agent 写一篇公众号文章，它应该先创建一个 `public_article` 类型的任务。然后系统会知道，这个任务的优先级不是「准，可执行，完整」。

而是「美，真，像」。

这三个字我还挺喜欢。

美。

真。

像。

公开文章得先有成品吸引力，但不能为了好看编。像不像某个人的风格，反而排在后面。

这个排序很关键。

因为如果你把「像」放第一，AI 很容易开始表演。表演语气，表演口头禅，表演一种它以为的人味。

读者会觉得，嗯，像是像了。

但不真。

不打动人。

如果你让它写技术文档，排序就完全变了。

技术文档是「准，可执行，完整，清晰，美，像」。

你看，准排第一。

这就对了。

技术文档最怕的不是不好看，最怕的是看起来很好，结果照着跑不通。

PRD 又是另一套。

「用，真，清晰，完整，美，像」。

第一位是用。

团队能不能拿它做决定，工程能不能拿它开工，测试能不能拿它验收。

这才是 PRD 的命。

**三，别把所有东西都扔进一个生成按钮**

这就是我为什么觉得，写作引擎不能只研究怎么写得更顺。

顺只是表面。

真正决定一篇内容质量的，是它被谁拿去干什么。

这件事听着很朴素，但我有时候觉得，很多 AI 产品就是没想清楚这一步。它们把所有文字都当成文字，于是所有任务都被塞进同一个生成按钮。

写文章，生成。

写 PRD，生成。

写文档，生成。

写邮件，生成。

慢慢就变成了一个万能文本榨汁机。

啥都能榨。

味道都差不多。

我这次做 PTCE，反而是想把这个榨汁机拆开。

不是所有水果都应该进同一个杯子。苹果有苹果的处理方式，咖啡豆有咖啡豆的处理方式。

你不能把咖啡豆扔进榨汁机，然后问它为什么不甜。

比较有意思的是，这套实现现在还很克制。

它没有假装自己已经全自动。

`content run` 现在只支持 `public_article`。如果你拿 PRD 或技术文档去跑，它会直接告诉你，这个类型现在没有 runner。

一开始我还觉得这是不是有点寒酸。

后来我反而觉得，这挺好。

因为它没有为了 demo 好看，把所有内容类型都接到一个半吊子的 runner 上。那种东西最危险，表面上什么都能跑，实际上一跑全是糊的。

现在的做法是，公开文章可以先有一个 MVP runner，其他类型先由 agent 生成 artifact，再通过 CLI 保存。

系统负责留痕。

模型负责判断和写作。

这两个边界先分清楚。

![图片]

我跟你说，这种边界在 agent 项目里太重要了。

因为模型很擅长临场发挥。临场发挥好处是快，坏处是它经常把系统状态也当成可以发挥的东西。

它觉得写完了。

但系统不知道。

它觉得这个就是最终稿。

但 artifact 没保存。

它觉得任务可以结束。

但 readiness 还是 draft。

这种混乱，短期看不出来。因为你只看最终文字，会觉得还行。

但你一旦要把它变成产品，就会发现不行。

产品不能只靠「我觉得写完了」。

产品得有状态。

得有记录。

得有可以 review 的产物。

**四，不性感的东西，才让系统能用**

这就是我说的工单感。

AI 写作以前像一个朋友在微信里帮你改稿。现在我想让它更像一个团队成员，在系统里接活、交稿、流转、验收。

这听起来可能没那么性感。

但我越来越相信，真正能让 AI 进入工作流的，都是这些不性感的东西。

权限。

状态。

版本。

artifact。

review checklist。

错误边界。

这些东西不会让 demo 看起来更炸。

但它会让一个系统不至于用两天就散架。

当然，现在这套东西还很早。mock server 还是文件存储，runner 也只跑了公开文章，四个 writing skill 也只是第一版骨架。

甚至我自己刚试的时候，agent 还会因为 skill 触发顺序，绕过 PTCE 直接去用别的写作 skill。

你看。

工单系统刚搭好，员工从窗户翻进来了。

但这个翻车也挺好。

它提醒我，下一步要补的不是更花的生成能力。

而是入口。

用户说，根据当前项目写篇文章，系统要能接住。用户说，帮我把这个实现整理成 PRD，系统要能接住。用户说，给工程团队产出技术设计，系统也要能接住。

接住以后，再分派给对应的 writing skill。

这才是真正的 agentic writing。

不是模型在聊天框里即兴写一段漂亮话。

而是它知道自己正在完成哪张工单。

知道自己该用哪套标准。

知道自己最终要交付什么。

也知道交付之后，系统里会留下痕迹。

![图片]

我现在有个很强的感觉。

未来我们评价一个 AI 写作产品，可能不能只看那篇文章写得好不好。

还要看它在写之前，有没有问对问题。

写的时候，有没有选对身份。

写完之后，有没有留下可以追踪的产物。

文字只是最上面的水花。

下面那套流转结构，才决定它能不能长期用。

说到底，写作不是排字。

写作是判断。

判断写给谁。

判断为什么写。

判断这篇东西应该追求好看、准确，还是可执行。

判断什么时候该停。

判断什么时候算完成。

而我给 AI 写作装这个工单系统，就是想把这些判断，从人的脑子里，慢慢搬到系统里。

这件事现在还很粗糙。

但我觉得方向是对的。

因为一个真正有用的写作 agent，不应该只是一个会说话的人。

它应该是一个会干活的人。

会领任务。

会看标准。

会交付。

会留痕。

会被 review。

听起来很不浪漫。

但说真的。

这可能才是 AI 写作真正开始变得靠谱的地方。
