# 我用 AI/agent 把 AI Homework Review 做出来的 20 天

这个项目最开始最容易让我产生错觉的地方，是页面已经能动了，但真实链路其实还没闭环。前台能看到结果，不代表中间的上传、状态流转、批量处理和数据落盘真的都接上了。

我后来越来越确认一件事：vibecoding 最有价值的地方，不是让 agent 替我做项目，而是让它帮我更快推进那些边界已经清楚的部分。真正决定项目能不能继续长大的，还是那些必须由人来收口的判断。

## 开场问题

一开始撞上的第一道墙，不是“要不要继续做”，而是 API 已经能出结果，却还没有把真实链路真正跑顺。

我把问题压回到最基本的验证路径上，重新去看输入、路由和状态流转，想确认卡住的地方到底是在页面层，还是在中间的调用链路上。

最麻烦的是，表面上每一步都像是通了，但一旦切到真实流程，批量处理就会在半路停住，结果也没真正稳定落到后端存储里。

所以这个项目真正的开场，不是“我做了一个 AI 网站”，而是“我先得把那个看起来像通了、其实没通的地方找出来”。

## 项目起步

起步阶段做的第一件事，不是先把页面堆完整，而是把目标压成一个能验证的最小闭环。

我先从最小入口搭起 API，让一次请求能从前端走到执行侧，再看状态是不是按预期回写。这里 agent 的作用很直接：它可以帮我快速把样板代码、页面壳子和一些重复实现铺出来，让我不用一直消耗在低价值重复劳动上。

但很快我就发现，agent 能把“东西写出来”，不等于它能替我判断“这条链路是不是已经真的跑通”。接口能返回，不代表任务能持续推进；页面能展示，不代表结果能稳定回传。

所以起步阶段真正重要的，不是先做出一个完整产品感，而是先证明这件事确实能被工程化地做起来。

## 真实链路打通

真正把链路打通的时候，问题就从“能不能开始”变成了“每一跳是不是都是真的”。

这个阶段我最依赖 agent 的地方，是帮我快速整理前端表单、接口调用、日志定位和一些中间实现；但我自己要盯的，是上传、读图、OCR、讲评生成这些节点到底有没有一个个落到真实服务上。

之前最容易出现的误判，是把某个局部成功当成整体成功。页面拿到了一段结果，不代表对象存储、OCR 和讲评链路都已经真实跑通；只要其中一段状态流转没对齐，最后的“成功”就只是一个更像样的演示。

所以这一步的意义，不是“终于出结果了”，而是“我终于知道每个结果是从哪里来的，也知道它会在哪里失败”。

## 项目复杂化

等项目继续往前走，复杂度马上就不是首页 demo 那么简单了。班级批量批改、异步任务、离线任务中心这些东西一进来，系统就不再只是单次执行，而是开始变成一串需要调度的动作。

这时候 agent 依然很好用，因为它很适合帮我拆页面、抽 hook、补胶水代码、整理任务流转相关的结构。但复杂化之后真正的难点，不再是“怎么把一个页面写出来”，而是“什么时候入队、什么时候回写、什么时候才算结束”。

并发一上来，原来顺手就能看懂的流程开始出现等待、重试和顺序错位。这里如果继续只靠 vibe 去推，项目很快就会进入一种“改一处，另外几处又开始飘”的状态。

所以从这个阶段开始，我对 agent 的用法也变了。不是继续无脑让它往前写，而是我先把任务边界、状态变化和回退路径想清楚，再让它去提速。

## 工程收口

真正决定这个项目有没有从 demo 长成系统的，不是前面那些“能跑起来”的瞬间，而是后面的工程收口。

离线任务中心、MySQL、部署方式、环境边界、VPC 访问，这些问题都不是那种“多试几次 prompt 就会自己变对”的类型。它们更像是工程现实：哪些依赖必须收在一起，哪些路径可以兜底，哪些地方必须给未来的排障留空间。

agent 在这里仍然有价值。它可以帮我快速整理配置、改路由、补实现、做局部重构。但真正的边界判断还是得我来做，因为只要这些边界没收紧，流程能跑也只是偶然，下一次变更或者部署时它就会重新变脆。

所以我后来越来越把 agent 当成推进器，而不是决策者。它负责提速，我负责收口。

## 最后的判断

回头看，这个项目最有价值的地方，不是“我用 AI 写了很多代码”，而是我更清楚地看见了 agent 在真实项目里的作用边界。

它非常适合帮我穿过空白期，快速把页面、接口、样板实现和局部重构推进起来；也很适合在需求已经明确的时候帮我压缩重复劳动，提高推进速度。

但只要问题进入到架构边界、状态机、真实链路、部署收口和质量判断，agent 就不能替我做决定。这个部分如果我自己不下场，项目最后只会停在一个更复杂、更会动的演示壳子上。

所以这 20 天真正留下来的，不是一套“万能 vibecoding 方法论”，而是一个更朴素的结论：agent 能帮我更快走到判断点，但真正让项目从演示走向可用的，还是那些必须由人来承担的判断和取舍。
