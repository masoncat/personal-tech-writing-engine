# 前端工程师在AI时代的出路，可能不是继续当一个更快的切图仔

事情是这样的。

这两年我身边的前端朋友，焦虑感明显变重了。

不是那种「明天公司就要把我裁了」的焦虑。

而是一种更诡异的感觉。

你打开 Cursor，开着 Claude Sonnet，旁边再放一个 Codex 或者 Claude Code，随手丢进去一个需求。

帮我写个表单。

帮我把这个组件拆一下。

帮我补个单测。

帮我看一下这个 hydration warning 怎么回事。

然后它哗啦啦给你吐出来一大堆代码。

有时候还真能跑。

你盯着屏幕看十秒钟，心里那个声音就冒出来了。

不是哥们，那我呢？？？

我以前觉得这个问题挺无聊的。

因为每次只要聊到 AI 会不会取代程序员，到头来都会变成两派互喷。

一派说完了完了，全完了。

另一派说不会不会，AI 就是工具。

说真的，这两个答案我都不太满意。

因为它们都太粗糙了。

前端工程师不会因为 Claude Code 会写 React 组件就原地消失，但如果一个前端工程师的全部价值，就是把设计稿翻译成 React 组件，那他确实会很难受。

这话听着有点刺耳但，我觉得这就是事实。

回到前端这块。

前端这个岗位一直有一个很尴尬的地方，它离用户最近，但经常被当成离业务最远。

产品说，这里加个按钮。

设计说，这里圆角 8px。

后端说，接口就这样了。

测试说，这里有 bug。

于是前端夹在中间，像一个大型人肉胶水层。

愚钝如我，以前也很容易把前端理解成「把东西做出来」。

做页面，做组件，做动画，做性能优化，做工程化。

当然这些都重要。

但到了 AI 这波之后，我越来越觉得，前端真正的出路，可能不是继续把「做出来」这件事卷到极致。

因为「做出来」正在变便宜。

Stack Overflow 2025 的开发者调查里有个数字很有意思，84% 的开发者已经在用或者准备使用 AI 辅助开发，专业开发者里每天用 AI 的比例也过半了。

但另一个数字更骚。

46% 的开发者说自己不信任 AI 输出的准确性。

你看，这就很真实。

大家一边疯狂用，一边疯狂不信。

这不就是我们现在每天写代码的状态吗。

让 Claude Code 先写。

写完自己看。

看完让测试跑。

跑完再骂它。

骂完继续用。

人类的复杂性，在这一刻被体现得淋漓尽致。。。

所以我觉得，AI 不是把前端工程师干掉。

它是把前端工程师身上最容易被标准化的那部分，快速压低价格。

你会写一个 modal，这件事不稀缺了。

你会写一个 table，这件事也不稀缺了。

你会把接口数据塞进页面，做 loading，做 empty，做 error，这件事正在变得越来越不稀缺。

那什么还稀缺？

我自己的感受是，三件事。

第一，理解用户到底在干嘛。

第二，理解系统为什么会烂。

第三，能把 AI 变成团队生产力，而不是变成代码垃圾制造机。

先聊第一个。

很多前端朋友可能会觉得，用户体验是设计师和产品的事。

我以前也会这么想。

但你真做过复杂一点的业务就知道，体验这东西到头来经常是前端兜底。

设计稿里一个输入框，看起来就是一个输入框。

但真实用户填的时候，可能复制了一个带空格的手机号，可能上传了一个 200M 的文件，可能在地铁里网络断断续续，可能点了提交以后页面没反应，于是他疯狂连点。

这些东西，Figma 里一般不会长出来。

后端接口文档里也不会长出来。

它们只会在真实世界里长出来，然后啪一下砸到前端脸上。

这时候 AI 能帮你写 debounce，能帮你写 retry，能帮你写一个看起来还不错的 toast。

但它不知道这个用户为什么急。

它不知道这个失败对业务来说有多严重。

它不知道这个页面背后，是一个刚下班的人在地铁里给孩子报课，还是一个运营同学半夜两点在后台改活动库存。

前端如果只看组件，就会觉得这些都是边角料。

但你如果看人，就会发现这才是前端最值钱的地方。

把一个界面变成一个人真的能完成的任务。

这玩意，AI 暂时还不太会。

再聊第二个，系统为什么会烂。

GitHub 2025 年把 Copilot coding agent 往 GitHub、VS Code、issue、pull request 这些地方塞，这事我觉得信号挺强的。

AI 已经不满足于在编辑器里补全几行代码了。

它开始进入工程协作链路。

这一下前端团队的差距会被拉得特别大。

一个代码库，如果组件边界清楚，测试能跑，design tokens 统一，接口契约明确，文档写得像人话，那 AI 进来以后是真的爽。

你给它一个明确 issue，它能顺着项目结构往下走。

你让它改一个组件，它知道测试在哪里。

你让它补一个状态，它知道哪些地方不能碰。

但如果一个代码库里，到处都是祖传组件，到处都是 any，到处都是复制粘贴的业务判断，测试一跑红一片，README 还停留在三年前。

那完了。

AI 不是来救你的。

AI 是来把这坨混乱复制得更快的。

我有时候觉得，AI 时代的工程化会重新变得性感。

以前很多人觉得工程化是洁癖，是架构师自嗨，是写业务的人懒得理的东西。

但现在不一样了。

因为 AI 特别吃上下文。

你不给它边界，它就乱猜。

你不给它测试，它就自信。

你不给它规则，它就发散。

你不给它一个干净的系统，它就把系统里最脏的那部分学得惟妙惟肖。

这尼玛就是大模型版本的近朱者赤，近墨者黑。

所以未来很值钱的前端，不一定是最会手写 webpack 配置的人。

而是那个能把工程环境整理到 AI 可以稳定工作的那个人。

他知道组件应该怎么分。

知道状态应该放在哪里。

知道哪些业务规则要写进测试。

知道哪些文档必须存在。

知道一个 PR 里哪些东西可以让 AI 改，哪些东西必须人来拍板。

这种人以前可能叫高级前端。

以后我觉得更像是 AI 协作时代的工程管理员。

不是管人，是管上下文。

再聊业务。

这块可能是最不性感，但也最重要的。

前端工程师如果一直把自己放在需求链路的末端，就会越来越被动。

因为需求越清楚，AI 越擅长。

你把一个已经拆好的卡片丢给 AI，它写得飞快。

你把一个已经确定的表格丢给 AI，它也能写。

你把一个已经设计好的流程丢给 AI，它至少能写个八九不离十。

真正难的是前面。

这个需求为什么要做？

用户到底卡在哪？

状态流是不是合理？

异常路径有没有想过？

这个页面上线之后看什么数据？

这个能力以后会不会被复用？

如果前端能参与这些问题，价值就不一样了。

你不再是「等别人想清楚以后帮忙实现」的人。

你变成了「一起把问题想清楚」的人。

这个身份变化很关键。

很多朋友可能会说，可我就是普通业务前端啊，公司也不让我参与那么早，我能怎么办。

我非常理解这种感觉。

不是每个人一上来都有机会做架构，也不是每个团队都允许前端插手业务。

但我觉得可以从小事开始。

你可以把自己负责的模块补一个关键测试。

你可以把某个祖传组件的边界写清楚。

你可以把一次线上问题整理成排查文档。

你可以在需求评审里多问一句，如果用户在这里失败了，下一步应该去哪。

你可以让 Codex 帮你把重复脚本整理出来，然后你来判断哪些能留下。

这些事都不宏大。

甚至有点琐碎。

但它们会慢慢改变你在团队里的位置。

你会从一个写页面的人，变成一个让系统更容易被理解的人。

而 AI 时代最缺的，恰恰就是可理解的上下文。

这两年大家很喜欢说前端已死。

我每次看到都挺无语的。

前端没死。

只是那种「只负责把 UI 写出来」的前端，会越来越难。

因为那部分工作，正在被 Claude Code、Codex、Cursor、GitHub Copilot 一点点吃掉。

但如果你往体验走，往系统走，往业务走，前端反而有一个很好的位置。

它离用户近，知道人怎么用。

它离工程近，知道代码怎么坏。

它离产品近，知道需求怎么变形。

这三个东西叠在一起，其实是很稀缺的。

回到最开始那个问题。

前端工程师在 AI 时代的出路是什么？

我现在的答案很朴素。

别只做一个更快的代码执行者。

去做那个能定义问题、整理上下文、判断结果的人。

代码会越来越便宜。

但判断不会。

上下文不会。

对真实用户的理解不会。

大时代啊，朋友们。

如果你还在学基础，那就继续学。HTML、CSS、JavaScript、浏览器、网络、性能，这些东西不会过时。AI 写错的时候，你得知道它错在哪。

如果你已经能独立做业务，那就别只盯着组件库和框架更新。去看用户路径，去看数据，去看异常状态，去看系统边界。

如果你已经带团队，那就赶紧把团队变成适合 AI 协作的样子。不是每个人都去背 prompt，而是让代码库、测试、文档、review、发布流程都能承接 AI 的速度。

说实话我也不确定这条路是不是适合所有人。

但我越来越确定一件事。

AI 时代不会奖励那个最会等待需求的人。

它会奖励那个最会把混乱变成上下文的人。
